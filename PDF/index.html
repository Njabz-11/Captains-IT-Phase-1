Task 1A: Scenario and Scope

Scenario

The Stellar Voyage Manager is a comprehensive space travel management system designed to streamline the booking, management, and maintenance of interplanetary and interstellar travel. In an era where space tourism and travel have become accessible to the public, there is a need for a robust system to manage the complexities of space travel logistics.

The application serves three primary user roles:

  • Travelers - Individuals booking space travel experiences
  • Administrators - Staff managing bookings, flights, and customer data
  • Maintenance Personnel - Technical staff responsible for spacecraft maintenance and readiness

Problem Statement

The space tourism industry faces several challenges in managing the complex logistics of space travel:

  • Coordinating spacecraft availability with travel demand
  • Ensuring proper maintenance schedules for spacecraft safety
  • Managing bookings and passenger information efficiently
  • Providing role-specific access to system features
  • Maintaining accurate records of destinations, spacecraft, and flight schedules

Project Scope

The Stellar Voyage Manager application addresses these challenges through a comprehensive system with the following scope:

In Scope:

  • User authentication and role-based access control
  • Flight booking and management
  • Spacecraft fleet management and maintenance tracking
  • Destination information management
  • Administrative reporting and analytics
  • User profile management

Out of Scope:

  • Payment processing integration
  • Real-time spacecraft tracking
  • Integration with external travel agencies
  • Mobile application version
  • Multi-language support

Project Goals

The primary goals of the Stellar Voyage Manager application are to:

  • Streamline the space travel booking process for travelers
  • Provide administrators with tools to efficiently manage flights and bookings
  • Enable maintenance personnel to track and schedule spacecraft maintenance
  • Maintain accurate records of all space travel activities
  • Ensure data security and integrity throughout the system

The application aims to enhance the space travel experience for all stakeholders while ensuring safety, efficiency, and customer satisfaction.

Task 2: Database Design

The Stellar Voyage Manager uses a Microsoft Access database with multiple linked tables to store and manage all application data. The database implements referential integrity to maintain data consistency and relationships between entities.

Database Schema

Table Relationships

The database consists of six main tables with the following relationships:

  • Users - Stores user information including credentials and roles
  • Flights - Contains flight details including destination, spacecraft, and schedule
  • Bookings - Links users to flights, representing passenger bookings
  • Spaceships - Stores information about the fleet of spacecraft
  • Maintenance - Records maintenance activities for spacecraft
  • Payments - Tracks payment information for bookings

Table Structures

Users Table

Field Name Data Type Description Constraints
UserID AutoNumber Unique identifier for user Primary Key
Name Text User's full name Not Null
Email Text User's email address Not Null, Unique
Password Text User's password Not Null
Role Text User's role (Traveler, Administrator, Maintenance) Not Null

Flights Table

Field Name Data Type Description Constraints
FlightID AutoNumber Unique identifier for flight Primary Key
DestinationID Text Destination of the flight Foreign Key (Destinations.DestinationID)
DepartureDate Date/Time Date and time of departure Not Null
ArrivalDate Date/Time Estimated date and time of arrival Not Null
SpaceshipID Text Spacecraft assigned to the flight Foreign Key (Spaceships.SpaceshipID)
SeatsAvailable Integer Number of available seats Not Null
Price Currency Price per seat Not Null
Status Text Flight status (Scheduled, Completed, Canceled) Not Null

Bookings Table

Field Name Data Type Description Constraints
BookingID AutoNumber Unique identifier for booking Primary Key
UserID AutoNumber User who made the booking Foreign Key (Users.UserID)
FlightID AutoNumber Flight being booked Foreign Key (Flights.FlightID)
BookingDate Date/Time Date and time booking was made Not Null
Status Text Booking status (Confirmed, Canceled) Not Null

Spaceships Table

Field Name Data Type Description Constraints
SpaceshipID AutoNumber Unique identifier for spacecraft Primary Key
Name Text Name of the spacecraft Not Null
Type Text Type of spacecraft Not Null
Capacity Integer Passenger capacity Not Null
Range Text Maximum travel range Not Null
Status Text Current status (Operational, Maintenance, Retired) Not Null

Destinations Table

Field Name Data Type Description Constraints
DestinationID AutoNumber Unique identifier for destination Primary Key
Name Text Name of the destination Not Null
Type Text Type of destination (Planet, Moon, Space Station, etc.) Not Null
Distance Text Distance from Earth Not Null
Description Memo Detailed description of the destination Not Null

Maintenance Table

Field Name Data Type Description Constraints
MaintenanceID AutoNumber Unique identifier for maintenance record Primary Key
SpaceshipID AutoNumber Spacecraft being maintained Foreign Key (Spaceships.SpaceshipID)
MaintenanceDate Date/Time Date and time of maintenance Not Null
Description Memo Description of maintenance performed Not Null
TechnicianID Text Technician who performed the maintenance Foreign Key (Users.UserID)

Payments Table

Field Name Data Type Description Constraints
PaymentID AutoNumber Unique identifier for payment Primary Key
BookingID AutoNumber Booking associated with payment Foreign Key (Bookings.BookingID)
Amount Currency Payment amount Not Null
PaymentDate Date/Time Date and time of payment Not Null
PaymentMethod Text Method of payment Not Null

Normalization

The database has been normalized to the Third Normal Form (3NF) to eliminate redundancy and ensure data integrity:

  • First Normal Form (1NF): All tables have a primary key, and all attributes contain atomic values.
  • Second Normal Form (2NF): All non-key attributes are fully functionally dependent on the primary key.
  • Third Normal Form (3NF): All attributes are directly dependent on the primary key and not on other non-key attributes.

Referential Integrity

Referential integrity is maintained through foreign key constraints that ensure:

  • A booking cannot exist without a valid user and flight
  • A flight cannot exist without a valid destination and spacecraft
  • A maintenance record cannot exist without a valid spacecraft and technician
  • A payment cannot exist without a valid booking

These constraints prevent orphaned records and maintain the consistency of the database.

function TAuthManager.Login(Email, Password: string): Boolean;
var
  DataModule: TStellarDataModule;
  Users: TUserArray;
  I: Integer;
begin
  Result := False;
  DataModule := TStellarDataModule.Create(nil);
  try
    Users := DataModule.GetUsers;
    for I := 0 to Length(Users) - 1 do
    begin
      if (Users[I].Email = Email) and (Users[I].Password = Password) then
      begin
        FCurrentUser := Users[I];
        FIsLoggedIn := True;
        Result := True;
        Break;
      end;
    end;
  finally
    DataModule.Free;
  end;
end;

function TStellarDataModule.GetUserBookings(UserID: string): TBookingArray;
var
  I, Count, Index: Integer;
begin
  Count := 0;
  for I := 0 to Length(FBookings) - 1 do
  begin
    if FBookings[I].UserID = UserID then
    begin
      Inc(Count);
    end;
  end;
  SetLength(Result, Count);
  Index := 0;
  for I := 0 to Length(FBookings) - 1 do
  begin
    if FBookings[I].UserID = UserID then
    begin
      Result[Index] := FBookings[I];
      Inc(Index);
    end;
  end;
end;

Dynamic Arrays

The application uses dynamic arrays to store collections of records. These arrays are used for in-memory data manipulation and are populated from the database:

procedure TStellarDataModule.LoadUsers;
var
  Query: TADOQuery;
  I: Integer;
begin
  Query := TADOQuery.Create(nil);
  try
    Query.Connection := ADOConnection;
    Query.SQL.Text := 'SELECT * FROM Users';
    Query.Open;
    
    SetLength(FUsers, Query.RecordCount);
    I := 0;
    
    while not Query.Eof do
    begin
      FUsers[I].UserID := Query.FieldByName('UserID').AsString;
      FUsers[I].Name := Query.FieldByName('Name').AsString;
      FUsers[I].Email := Query.FieldByName('Email').AsString;
      FUsers[I].Password := Query.FieldByName('Password').AsString;
      FUsers[I].Role := TUserRole(Query.FieldByName('Role').AsInteger);
      
      Inc(I);
      Query.Next;
    end;
  finally
    Query.Free;
  end;
end;

function TStellarDataModule.LoadDestinationDetails(DestinationID: string): string;
var
  FileName: string;
  FileContents: TStringList;
begin
  Result := '';
  FileName := ExtractFilePath(Application.ExeName) + 'Data\Destinations\' + DestinationID + '.txt';
  
  if FileExists(FileName) then
  begin
    FileContents := TStringList.Create;
    try
      FileContents.LoadFromFile(FileName);
      Result := FileContents.Text;
    finally
      FileContents.Free;
    end;
  end;
end;

These text files provide a way to store rich, detailed information that would be cumbersome to store in the database. They also allow for easy updates to descriptive content without modifying the database structure.
